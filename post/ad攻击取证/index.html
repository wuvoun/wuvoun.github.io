Active Directory 攻击取证

检测 NTDS.dit 转储

  • 事件 7036

    用来记录 Windows 服务状态的变化, 如果在日志里发现一些敏感服务被异常启动/停止（如 Remote Registry, WinRM, Telnet 等），就要警惕是否有人在进行横向移动或远程控制。

使用 ntdsutil 或 Windows 内置功能

  • 转储指令
ntdsutil "activate instance ntds" "ifm" "create full C:\ntdsbackup" quit quit
  • 取证

Security.evtx → 4688 (Process Creation)

存在进程 ntdsutil.exe 存在指令 ntdsutil “activate instance ntds” “ifm” …

在 IFM 创建数据库副本的过程中，系统为了避免复制时 NTDS.dit 处于写入状态，后台会通过 NTDS 服务调用 Windows 的快照机制 来生成一个一致性的副本，然后再从快照中提权ntds, 所以会从日志中发现Volume Shadow Copy

使用 esentutl

  • 转储指令

    esentutl /y C:\Windows\NTDS\ntds.dit /d C:\evidence\ntds_copy.dit

    （/y = copy，/d = 转储路径，通常还加 /vss 参数）

  • 取证检查

    • Security.evtx → 4688 出现

      esentutl.exe /y C:\Windows\NTDS\ntds.dit ...

    • Security.evtx → 4656/4663（访问 C:\Windows\NTDS\ntds.dit 文件）

VSS 卷影拷贝

创建卷影快照时，卷影复制服务会验证权限 ，使用计算机帐户并枚举用户组。

  • 转储指令

    vssadmin create shadow /for=C:
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\NTDS\ntds.dit C:\tset\
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\System32\config\SYSTEM C:\test\

    或用 wmic shadowcopy call create Volume=C:\ 创建影子盘

  • 取证检查

    • System.evtx → 7036 (VSS 服务启动/停止)
    • Application.evtx / System.evtx → 8193/8224 (VSS snapshot 创建)
    • Security.evtx → 4688 出现 vssadmin.exe 或 wmic shadowcopy

分析 Microsoft-Windows-NTFS 事件日志，筛选事件 ID 4、9、10、300 和 303，这将过滤只有 NTFS 卷装载和卸载事件

事件 ID 描述 备注
4 卷被装载（Volume Mounted） 当 NTFS 卷被系统挂载时触发。
9 卷被卸载（Volume Unmounted） 当 NTFS 卷被卸载时触发。
10 卷被扫描或检测（Volume Detected/Scanned） 系统检测到卷的存在或扫描卷。
300 卷创建快照（Volume Snapshot Created） 相关于 VSS（卷影复制服务）。
303 卷影复制完成或状态改变（Shadow Copy Event） 卷影复制的开始/结束状态。

DCSync

  • 转储指令 (Mimikatz)

    mimikatz.exe "lsadump::dcsync /domain:域 /all" "exit"

  • 转储指令 (Impacket)

    secretsdump.py <domain>/<user>@<dc-ip>

  • 取证检查

    • Directory Service.evtx → 4662 对象访问，属性包含
      • DS-Replication-Get-Changes
      • DS-Replication-Get-Changes-All
    • Security.evtx → 4624 (Logon Success) 来自攻击者机器

Kerberoasting取证

当域用户想要向服务进行身份验证时，拿着TGT会向 DC 进行身份验证， DC 验证了这个TGT， 会返回一个使用服务帐户密码的 NTLM 哈希加密的 TGS，以便用户将其传递给服务时，外面可以解密该 TGS。

Kerberoasting 正是将加密的 TGS 脱机以尝试暴力破解，如果密码较弱，则破解可能会成功。

当用户或服务帐户请求访问另一个服务时，向 KDC请求一个服务票据 (TGS)。这个操作的事件日志为

4769

常用工具

  • Rubeus
  • impacket-GetUserSPNs
  • Invoke-Kerberoast（PowerSploit/PowerView）
  • Kerbrute

AS-REP-Roasting取证

Kerberos 身份验证流程中，用户首先向 DC 进行身份验证请求TGT，DC经过验证，验证通过然后向用户发送TGT，该票证的一部分使用帐户的 NTLM 哈希加密。

而要请求TGT，DC会验证你是不是这个用户，但是，用户帐户可以设置一个名为“不需要 Kerberos 预身份验证”（ UF_DONT_REQUIRE_PREAUTH ）的选项。如果设置了此选项，用户在请求票证之前无需进行身份验证。对这部分hash的爆破称为 AS-REP-Roasting。

  • 针对 不需预身份验证的用户（UF_DONT_REQUIRE_PREAUTH）。
  • 检测：
    • Security.evtx → 4768 (TGT 请求)，PreAuthType = 0

执行 AS-REP-Roast 攻击，攻击者请求TGT。这将生成一个 *4768* 事件

以下为PreAuthType 字段，Kerberos 预身份验证（Pre-Authentication）类型代码

值 名称 含义 在 AD 中常见情况
0 无需预身份验证 客户端可以直接请求 TGT，不需额外验证 AS-REP Roasting 攻击可利用
2 PA-ENC-TIMESTAMP 标准的基于时间戳的密码验证方式（最常见） ✅ AD 默认用
11 PA-ETYPE-INFO KDC 在 KRB-ERROR 中提示客户端使用哪个密钥加密时间戳 ❌ AD 中几乎不用
15 PA-PK-AS-REP_OLD 旧版智能卡登录认证 已废弃
17 PA-PK-AS-REP 智能卡身份验证（新版） 部分环境有，但 AD 里极少
19 PA-ETYPE-INFO2 与 11 类似，但支持更多加密类型 ❌ AD 中几乎不用
20 PA-SVR-Referral Info 用于 KDC 推荐票（跨域场景） 常见于林/信任关系
138 PA-ENC-CHALLENGE Kerberos Armoring (FAST) 登录，提升安全性 从 Win2012 DC + Win8 客户端 开始支持

LLMNR 中毒攻击

LLMNR（Link-Local Multicast Name Resolution）是 Windows 环境下的一种名称解析协议，主要在 DNS 查询失败时作为后备机制使用。它通过 UDP 5355 端口 在本地子网内进行广播/组播请求

通过流量分析或事件监控来找出恶意 IP

  • Wireshark 过滤器：
udp.port == 5355
llmnr
nbns && nbns.flags.response == 0
  • 多台主机响应同一个查询 → 可疑
  • 受害者会发出 NetNTLMv2 哈希

重点关注 LLMNR 响应报文（Standard query response） 如果出现 多台主机都在响应同一个查询，就很可疑

将这个 恶意 IP 继续反查得到机械名：

  • NetBIOS Name Service (NBNS) 流量（UDP 137）
  • SMB 会话建立（协议里通常带 NetBIOS 名/主机名）
  • mDNS/ARP 也可能泄露设备名
  • DHCP 日志

重组hash

当受害者机器认为攻击者就是 自己要访问的服务，于是尝试访问：

  • 可能是 SMB (445端口)
  • 也可能是 HTTP (80/443)

在访问过程中，Windows 会自动尝试使用 NTLM 认证。

在 NTLM 认证流程里，受害者会发送 NetNTLMv2 哈希（基于用户密码加密的 challenge-response），攻击者工具就能捕获到这个哈希

  • 攻击者可以尝试 离线破解哈希（暴力/字典），得到明文密码。
  • 或者直接做 NTLM Relay 攻击，把这个认证转发到真正的服务器，获取访问权限。

常看到 NetNTLMv2 Hash 长这样

username::DOMAIN:ServerChallenge:NTProofStr:Blob
  • ServerChallenge = 服务器发来的 8 字节随机数
  • Blob = 客户端生成的一些字段（时间戳、客户端随机数、域信息等）
  • NTProofString 就是用来证明客户端确实知道正确密码哈希的签名。

NTLM 中继攻击

攻击者将受害者的身份验证请求“转发”到另一台目标服务器，实现自己以受害者身份认证。 对 NetNTLMv2 响应进行捕获的工具常见的有 Responder, Metasploit

  • PCAP 分析

    • 使用 Wireshark 分析 SMB、LDAP NTLM 流量

    • 搜索 NetNTLMv2 响应包：

      ntlmssp

    • 确认攻击者 IP、受害者 IP、目标服务器 IP

  • 日志关联
    • 查看 Event ID 4776（域控制器上的 NTLM 验证事件）
    • 对比 Source IP 和登录用户，检查异常登录或中继模式

流量包主机识别

NetBIOS (NBNS) 刷新数据包，其实就是 NBNS 名称刷新或广播请求，它通常出现在 Windows 主机启动或网络变化时，用来通知或更新网络上其他主机的名称/IP 对应关系 NBNS 刷新包本身就是主机告诉网络 “我的名字还是这个，IP还是这个”，所以抓到刷新包就能直接看到对应 IP，非常适合用来快速确定主机 IP。

NBNS 刷新包的特点

  • 目标地址：通常是广播 255.255.255.255，或者子网广播
  • 包的作用：更新 NBNS 缓存，保持名称-IP 对应关系
  • Source IP：就是发送刷新包的主机 IP
  • 在抓包中找 IP：

    • Wireshark 里过滤：

      nbns && nbns.flags.response == 0

      这通常显示注册或刷新请求

    • 或者只看响应：

      nbns && nbns.flags.response == 1

      响应里有 Owner IP，就是主机实际 IP

什么是 MFT文件

可以把 $MFT 理解为 NTFS 的“目录数据库” 或者 索引表

作用：

  • 存储每个文件/目录的元数据，包括：
    • 文件名
    • 文件大小
    • 时间戳（创建/修改/访问）
    • 权限（ACL）
    • 文件数据的磁盘存放位置（簇地址）
  • NTFS 通过 $MFT 来快速查找和管理文件。

意义：

即使文件被删除，$MFT 中可能还保留残留信息（如文件名、大小、时间戳），所以在取证分析中非常重要。

工具处理 ：

将*$MFT 转换为csv文件*

MFTECmd.exe  -f '$MFT'  --csv  mft.csv

什么是Prefetch 文件

Prefetch 文件是 Windows 系统里一种 应用程序运行痕迹文件

  • Windows 从 Windows XP 开始引入 Prefetch 功能，目的是 提升应用启动速度。
  • 当一个程序第一次运行时，系统会生成一个 .pf 文件，记录程序加载过的 DLL、文件路径、运行次数等信息。
  • 下一次运行时，系统会根据 .pf 文件提前加载所需数据，从而加快启动。

命名规则：

<程序名>-<哈希值>.pf

分析prefetch文件可以知道：

  • 某个可执行文件是否运行过？
  • 运行过多少次？
  • 最后一次运行的时间？
  • 运行时加载了哪些 DLL/文件？

分析工具

PECmd.exe -f C:\Windows\Prefetch\pf文件
